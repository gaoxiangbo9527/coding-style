属性名使用 camelCase 命名风格。
    PSR-12 没有指明属性名的命名风格，这里补充。

当为类常量声明可见性时，把可见性关键字放在 const 关键字前面，例如：
    public const STATUS_OK = 1;

行长软限制依然保持为 120 个字符，但建议行长超过 120 个字符才换行。
    PSR-12 中提到软限制是 120 个字符，但建议行长超过 80 个字符才换行。
    所谓 80 列换行的标准已经在慢慢过时，严格追求它意义不大。

不要省略 PHP 代码块中最后一条语句的分号。

包名和命名空间名
    包名
        包名应该由供应商名和项目名组成的。Composer 和 Github 上的项目几乎都是这样。
        供应商名和项目名请按照下方各自的规则书写。
    供应商名
        供应商名可以是公司名或用户自己的网名。
        由小写字母、数字和连字符组成。
        单词之间使用连字符分隔；但应尽可能短和明确，从而尽量不用连字符。
        可以包含数字，但不推荐。
        根据需要，可以考虑在文件系统中建立同名的目录，个人建议这么做。
    项目名
        项目名可以根据项目自行命名。
        由小写字母、数字和连字符组成。
        单词之间使用连字符分隔；但应尽可能短和明确，从而尽量不用连字符。
        可以包含数字，但不推荐。
        必须在文件系统中建立同名的目录。
        项目名要么应该普通，要么应该特别；普通时要借助供应商名保持唯一，特别时不借助供应商名就可以保持唯一。
    命名空间名
        推荐由供应商名和项目名组成命名空间前缀，但应将供应商名和项目名转换成 PascalCase 风格。
        如果项目名可以保持唯一，可以考虑从命名空间前缀中去除供应商名，从而避免命名空间名太长。
        子命名空间应该使用 PascalCase 风格，子命名空间对应的目录名应该使用 PascalCase 风格。
        为了避免命名空间名太长，应认真规划子命名空间的层级，不要有太多层级，建议最多两个子命名空间层级。
    非命名空间对应的文件和目录
        没有统一的规范，多参考优秀的框架和库的惯例。更重要的是团队或者项目内要统一。

文件和目录名单复数
    如果能参考惯例就参考惯例，如果没有惯例可以参考则建议使用单数形式。更重要的是团队或者项目内要统一。

字面量相关
    布尔型字面量
        布尔型类型有两个字面量，即大小写不敏感的常量 TRUE 和 FALSE 。虽然它们都是常量，但是建议使用全小写形式的 true 和 false 。
    整型字面量
        二进制、十六进制表示法的前缀建议使用全小写字母形式，即使用 0b 和 0x ，而不使用 0B 和 0X 。
    浮点型字面量
        当使用科学计数法表示浮点型字面量时，使用 e ，而不使用 E 。
        浮点型有一个特殊的字面量，即大小写不敏感的常量 NaN 。虽然它是常量，但是建议使用全小写形式的 nan 。
    字符串字面量
        当使用 Heredoc 表示字符串字面量时，建议总是给 Heredoc 的开始标识符加上双引号，从而和 Nowdoc 形成照应。
    数组字面量
        建议使用 [] 语法，而不是 array() 。
        最后一个数组元素后的逗号是可选的并且可以被省略。
            对于单行数组，通常会省略尾随逗号，即 [1, 2] 优先于 [1, 2,] 。
            对于多行数组，通常会使用尾随逗号，因为这么做可以更容易地在数组末尾添加新元素。
    null 类型字面量
        null 类型只有一个字面量，即大小写不敏感的常量 NULL 。虽然它是常量，但是建议使用全小写形式的 null 。
